# Repository Guidelines

## Project Structure & Module Organization

FreeTyping is a static Vite + React + TypeScript app. Source code lives in `src/`.
Main app state and shell code are in `src/app/`. UI components are grouped under
`src/components/`, with one folder per larger component. Top-level input modes
are System and Overlay; overlay behavior is selected by keyboard layout. Input
logic is in `src/input/`, including Chinese Pinyin under
`src/input/modes/chinesePinyin/`. Layout data is in `src/keyboard/layouts/`.
Reusable hooks live in `src/hooks/`, mouse models in `src/mouse/`, and shared
helpers in `src/utils/`. CSS is split across `src/styles/`. Browser-level checks
are in `tests/`; generated screenshots go to `test-artifacts/`. Static
dictionary data is in `public/data/`, generated by
`scripts/build-pinyin-dictionary.mjs`.

## Build, Test, and Development Commands

- `npm run dev`: start the local Vite dev server.
- `npm run build`: type-check and build static output into `dist/`.
- `npm run preview`: serve the production build locally.
- `npm run test`: run Vitest unit tests.
- `npm run test:browser`: run headless browser smoke checks and screenshots.
- `npm run release:check`: run unit, browser, accessibility, bundle, and static-host checks.
- `npm run build:pinyin-dictionary`: regenerate `public/data/pinyin-dictionary.json`.

## Coding Style & Naming Conventions

Use TypeScript and React function components. Keep components PascalCase
(`TextEditor.tsx`) and hooks camelCase with `use` prefix. Prefer small, focused
modules and local helpers over large shared abstractions. Use 2-space
indentation, single quotes, and existing CSS class naming patterns such as
`text-editor__body`. Keep comments rare and useful.

## Testing Guidelines

Use Vitest for unit tests with `*.test.ts` files near the module under test.
Browser checks are plain Node scripts in `tests/`. Add unit coverage for parsing,
state, and utility logic; add browser smoke coverage for user-visible workflows,
layout-sensitive behavior, and persistence. Run `npm run release:check` before
publishing tested work.

## Commit & Pull Request Guidelines

Commit messages use short imperative summaries, for example `Add about menu item`
or `Archive v0.1.0 plans`. Pull requests should describe the user-facing change,
list tests run, and include screenshots when layout changes. Mention related
issues when applicable.

## Release & Deployment Notes

`main` is the development branch and may contain untested work. Tested static
builds are published from `deploy`, which is the GitHub Pages source branch.
Official release history is tracked in `plan/version.md`; completed plans are
archived by release, such as `plan/0-1-0/` and `plan/0-2-0/`.
